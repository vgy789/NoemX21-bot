# Документация Webhook API

## Обзор

Webhook API позволяет внешним приложениям проверять, зарегистрирован ли пользователь Telegram как участник «Школы 21» в системе noemx21-bot.

## Аутентификация

Все запросы к API должны содержать API-ключ в заголовке `X-Secret`. Пользователи могут сгенерировать свой API-ключ через меню настроек бота:

1. Откройте бот в Telegram.
2. Перейдите в: **Настройки → API → API Token**.
3. Нажмите «Сгенерировать новый» (Generate new), чтобы создать ключ.
4. Скопируйте созданный ключ (формат: `noemx_sk_...`).

**Примечание:** Генерация нового ключа аннулирует все предыдущие ключи.

## Эндпоинт

### POST /api/v1/webhook/register

Проверяет, зарегистрирован ли пользователь Telegram с указанным `external_id` под конкретным логином «Школы 21».

#### Запрос

**Заголовки:**

* `X-Secret`: Ваш API-ключ (обязательно).
* `Content-Type`: application/json.

**Тело запроса:**

```json
{
  "external_id": "123456789",
  "login": "student-login"
}

```

**Параметры:**

* `external_id` (string): ID пользователя в Telegram.
* `login` (string): Логин «Школы 21» для проверки.

#### Ответ

**Успех (200 OK):**

```json
{
  "registered": true
}

```

**Поля ответа:**

* `registered` (boolean):
* `true`, если пользователь Telegram зарегистрирован и привязан к указанному логину.
* `false`, если пользователь не зарегистрирован или привязан к другому логину.



#### Ошибки

**401 Unauthorized:**

* Отсутствует заголовок `X-Secret`.
* Неверный API-ключ.

**405 Method Not Allowed:**

* Метод запроса отличен от POST.

**400 Bad Request:**

* Некорректный JSON в теле запроса.

**500 Internal Server Error:**

* Ошибка базы данных или иная серверная ошибка.

## Примеры использования

### cURL

```bash
curl -X POST http://localhost:8081/api/v1/webhook/register \
  -H "X-Secret: noemx_sk_your_api_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "external_id": "123456789",
    "login": "student-login"
  }'

```

---

## Архитектура

* **Порт**: 8081 (по умолчанию).
* **Аутентификация**: API-ключи хранятся в базе данных в виде SHA256-хэшей.
* **Формат ключа**: Префикс `noemx_sk_`, за которым следуют 64 шестнадцатеричных символа.
* **Хранение ключей**: В базе данных хранится только хэш; пользователям показывается префикс для идентификации ключа.
