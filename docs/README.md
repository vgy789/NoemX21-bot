# Техническое Задание (ТЗ) на разработку Telegram-бота для Школы 21

## 1. Цель проекта
Разработка Telegram-бота для студентов Школы 21, предоставляющего единый интерфейс для взаимодействия с сервисами школы (бронирование, проверки, статистика) и упрощения коммуникации внутри комьюнити.

## 2. Функциональные требования
Функциональные сценарии и UIX описаны в YAML-спецификациях. Эти файлы — **источник правды** по пользовательским потокам (экраны/состояния/ошибки/тексты/кнопки), на основе которых проектируются хендлеры бота и структура данных.

### 2.1. Спецификации Flow (`docs/specs/flows/*.yaml`)
- `docs/specs/flows/main_menu.yaml` — главное меню: навигация, доступные разделы, базовые команды.
- `docs/specs/flows/registration.yaml` — регистрация и привязка аккаунта: верификация, статусы, ошибки, повторная попытка.
- `docs/specs/flows/settings.yaml` — настройки пользователя: язык, приватность, токены/интеграции, уведомления.
- `docs/specs/flows/statistics.yaml` — статистика: отображение прогресса и метрик (XP/PeerPoints/Coins и т.п.), сравнения/срезы (если предусмотрено).
- `docs/specs/flows/booking.yaml` — бронирование ресурсов кампуса: поиск, слоты, создание/отмена, ограничения.
- `docs/specs/flows/library.yaml` — библиотека: поиск книг, наличие, бронирование, напоминания/уведомления.
- `docs/specs/flows/reviews.yaml` — peer-review: запросы на проверку, фильтры, отклики, статусы.
- `docs/specs/flows/clubs.yaml` — клубы/сообщества: каталог, карточки, вступление/контакты, создание (если предусмотрено).
- `docs/specs/flows/admin.yaml` — админские сценарии: рассылки, модерация, привязка топиков/групп, guard-механики.

### 2.2. Краткий перечень функций (по разделам)
- **Регистрация/профиль**: `registration.yaml`, `settings.yaml`, `statistics.yaml`
- **Сервисы кампуса**: `booking.yaml`, `library.yaml`
- **Комьюнити/обучение**: `reviews.yaml`, `clubs.yaml`
- **Администрирование**: `admin.yaml`

## 3. Технические требования и Стек
Технологический стек определяется требованиями к производительности, поддерживаемости и простоте деплоя.

### 3.1. Технологический стек
- **Язык:** Go 1.25+
- **База данных:** PostgreSQL 18+ (хранение данных, очередей, кэша).
- **ORM/Query Builder:** `sqlc` (генерация типобезопасного кода из SQL).
- **Миграции:** `golang-migrate`.
- **Telegram API:** `gotgbot/v2`.
- **Внешние интеграции:**
    - School 21 API (валидация, статистика).
    - Rocket.Chat API (поиск пользователей, отправка сообщений).
    - Документация по внешним API и примеры запросов: `docs/api/` (reference для интеграций, не Public API бота).

### 3.2. Архитектура и Инфраструктура
Системная архитектура визуализирована с использованием модели C4:
- **Контекст, Контейнеры и Компоненты:** `docs/c4/`

Системные требования и эксплуатационные политики детализированы в `docs/specs/system/*.yaml`.

#### 3.2.1. Системные спецификации (`docs/specs/system/*.yaml`)
- `docs/specs/system/security_policy.yaml` — политика безопасности: PII, криптография, доступы, угрозы/контрмеры.
- `docs/specs/system/secrets_management.yaml` — управление секретами: хранение, ротация, окружения, CI.
- `docs/specs/system/hardening.yaml` — hardening: минимизация привилегий, настройки ОС/контейнеров, рекомендации по защите.
- `docs/specs/system/network.yaml` — сеть и периметр: reverse proxy, TLS, изоляция, правила доступа.
- `docs/specs/system/rate_limiting.yaml` — rate limiting/защита от злоупотреблений: лимиты, ключи, реакции.
- `docs/specs/system/audit_logging.yaml` — аудит/логирование: события, форматы, хранение, требования к неизменяемости.
- `docs/specs/system/monitoring.yaml` — мониторинг/алертинг: метрики, логи, healthchecks, SLO (если задано).
- `docs/specs/system/operations.yaml` — эксплуатация: runbook-практики, регламенты, обслуживание, доступы.
- `docs/specs/system/ci_cd.yaml` — CI/CD: пайплайны, проверки, окружения, релизы/деплой.
- `docs/specs/system/disaster_recovery.yaml` — DR: бэкапы, RTO/RPO, сценарии восстановления.
